Public Pages — Static Content Audit

All pages under frontend/src/app/(public)/ — classified by how much hardcoded / static content remains vs. DB-driven sections.

Legend

✓ DB-DrivenPage fetches every section/content from the API. Nothing hardcoded.
⚠ MixedPage queries the API but also contains hardcoded fallback sections rendered as static React components that bypass the DB.
✗ StaticPage layout / copy is entirely hardcoded in TSX. Only data rows (news cards, event cards) come from the API.

Home Page — Most Critical

RouteFileIssueStatus
/ (public)/page.tsx Queries DB site-sections first, but every section is implemented as a static local React function:
  • HeroSection() — title, subtitle, buttons, SVG art all hardcoded
  • OurFocusAreasSection() — 3 hardcoded focus-area cards (Research, Education, Sustainability)
  • JoinOurCommunitySection() — CTA copy hardcoded
  • TestimonialsSection() — 3 hardcoded fake testimonials (Dr. Ahmad K., Fatima M., Prof. Omar S.)
  • ContactFormSection() — form that calls e.preventDefault() only — never submits to the API
  • NewsletterSection() — subscribe form uses setTimeout(1000) fake delay — never calls the API
  • StatisticsSection() — stats array (15+ years, 500+ members, 200+ pubs, 50+ events) hardcoded constants
  • CTAsSection() — "Become a Member" / "Contact Us" CTA copy hardcoded
  • AboutSection() — full about text hardcoded (never rendered in the final render tree — dead code!)
When DB site-sections exist, the page still renders the same hardcoded components — the DB sections only choose which hardcoded component to show, they do not feed data into it. 		✗ Static façade

What is Legitimately Static (should stay)

PageWhy static is fine
Loading skeletons, error states, "not found" banners UI chrome — not content. Correct to be in code.
Pagination controls, search form widget Behaviour, not content.
Tailwind classes, layout structure Presentation layer, belongs in code.
/page/[slug] — back-to-home link "Page Not Found" text Fallback UI, acceptable in code.

Recommended Actions (Priority Order)

#ActionImpact
1 Home page — wire real data into existing section components
Hero: read title/subtitle/buttons from DB site-section config JSON.
TestimonialsSection: call /api/public/testimonials (or use PageSectionRenderer).
StatisticsSection: call an API or at minimum make constants configurable via admin.
ContactFormSection & NewsletterSection: wire to /api/public/contact and /api/public/newsletter/subscribe. 		High — fake form / fake stats visible to users
2 Replace AboutSection() dead-code on home page
It is defined but never rendered in the current render path. Either delete it or plug it into the section loop. 		Medium — dead code / confusion
3 News / Events / Jobs / Members / Search — replace hardcoded hero banners with a DB page header
The cleanest approach: add a "page-header" entry in pages table for each route (slug = "news-page-header" etc) and render it via PageSectionRenderer. Alternatively keep a simple API call for the page title/subtitle only. 		Medium — admin cannot change headings without a deploy
4 Members page — load MEMBERSHIP_TYPES and SPECIALIZATIONS from API
Hardcoded filter arrays mean new membership types added in the DB won't appear as filter options. 		Medium — filter out-of-sync with DB
5 Jobs / Events pages — load EVENT_TYPES filter from API
Same issue as members — if new types are added to the DB they won't appear as filters. 		Low–Medium

Summary: The 8 CMS page-builder pages (/about, /board, /contact, etc.) are fully DB-driven and require no changes. The 8 data-list pages and the home page still have significant static hardcoded content — most critically the home page contact/newsletter forms which do not submit to the API, and the hardcoded testimonials/statistics that admins cannot update without a code deployment.
