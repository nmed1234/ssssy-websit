Page Builder — Full Refactor Plan

Architectural redesign: flat sections → nested tree with true layout containers, live editing, and data-driven rendering.

1. Current Problems (Audit)

Architecture — Flat list, no nesting

The DB stores a list of PageSection rows, each with a flat data JSON blob. There is no parent/child relationship. Adding a component "inside" a section is impossible at the data level.

Two disconnected property editors

The admin page (admin/pages/[id]/page.tsx) uses AdvancedPropertyEditor. The standalone PageBuilder component uses a completely different PropertyEditor. They are not in sync — changes to schemas or features in one don't exist in the other.

Added component not shown / not editable

When you drag a component from the palette, createDefaultSection creates a shell with data: { title:"", content:"" }. The BuilderCanvas renders it via PageSectionRenderer which routes to a hardcoded React component (like AboutVisionMissionSection) that ignores the empty data entirely — so nothing visible appears. The AdvancedPropertyEditor panel has generic tabs (Content / Layout / Style / BG / Events) but no field-schema awareness for most component types, so fields don't map to the actual data structure.

Hardcoded page sections

Components like AboutVisionMissionSection, AboutHeroBanner, etc. are static React components with hardcoded text. The builder cannot edit them because their data is not driven by section.data.

Pick-item overlay is a heuristic hack

The canvas overlay divides the section into N equal columns and guesses which "item" the user clicked. It has no relation to actual DOM positions of rendered children.

No layout engine

There is no concept of a Row containing Columns, or a Grid with children. All components sit in a linear list at the page level only.

2. Proposed Architecture

2.1 Data Model — Nested Tree

Every node in the page is a block. Blocks can have children. This replaces the flat PageSection[] list with a recursive tree stored as a single JSON document per page.

Page (stored in DB as pages.layout_json) └── Block[] (the root children) ├── Block { id, type:"section", props:{}, children:[] } │ ├── Block { id, type:"row", props:{gap:"1rem"}, children:[] } │ │ ├── Block { id, type:"column", props:{width:"1/3"}, children:[] } │ │ │ └── Block { id, type:"heading", props:{text:"Our Vision"} } │ │ └── Block { id, type:"column", props:{width:"2/3"}, children:[] } │ │ └── Block { id, type:"paragraph", props:{text:"..."} } │ └── Block { id, type:"card-group", props:{columns:3}, children:[] } │ ├── Block { id, type:"card", props:{title:"Vision",...} } │ ├── Block { id, type:"card", props:{title:"Mission",...} } │ └── Block { id, type:"card", props:{title:"Objectives",...} } └── Block { id, type:"divider", props:{} }
FieldTypeDescription
idstringUUID, stable across edits
typestringComponent type key (e.g. "section", "heading")
propsobjectAll editable properties (text, urls, colors, layout) in one flat object per block
childrenBlock[]Nested child blocks; undefined/[] for leaf nodes
visibilitystringALWAYS / HIDDEN / LOGGED_IN / etc.

DB strategy: Add a layout_json TEXT column to the pages table. Keep page_sections for backwards compatibility. The new builder writes/reads only layout_json.

2.2 Component Type Taxonomy

Containers (have children)

  • section
  • row
  • column
  • grid
  • flexbox
  • tabs (children = tab panels)
  • accordion (children = items)
  • card-group (children = cards)

Leaf Blocks (no children)

  • heading
  • paragraph
  • image
  • video
  • button
  • card
  • divider / spacer
  • rich-text
  • stat-item, team-member, etc.

Dynamic Blocks

  • news-feed
  • events-feed
  • jobs-feed
  • members-feed
  • contact-form
  • newsletter-form
  • map

2.3 Props Schema Registry

A single BLOCK_SCHEMA registry maps every block type to its editable props. Each prop has a type (text, textarea, number, color, select, toggle, url, richtext), a label, and an optional group (Content / Layout / Style). The property panel is auto-generated from this schema — no hardcoding needed.

2.4 Renderer — Props-Driven, No Hardcoded Sections

A single BlockRenderer recursively renders any block tree. Container blocks render their children. Leaf blocks render from their props directly. No more AboutVisionMissionSection.tsx, AboutHeroBanner.tsx, etc. — all replaced by generic blocks with specific default props.

Key canvas interactions

  • Click any block → selects it, shows blue outline, loads its props in right panel
  • Double-click a container → enters the container (breadcrumb updates), can select children
  • Drag from palette into a container → adds new child at drop position
  • Drag handle on block → reorder within same parent
  • Escape → navigate up one level in breadcrumb
  • Inline text editing → click on a heading/paragraph to edit text in-place

4. Property Panel Design

The right panel shows tabs driven by the selected block's schema:

TabWhat it shows
ContentText fields, textareas, rich-text, URLs, select — the semantic content of the block (title, body, href, etc.)
LayoutWidth, padding, margin, alignment, columns, gap — only for container blocks
StyleBackground color/image/gradient, text color, font-size, border, shadow, border-radius
AdvancedVisibility rule, animation, custom CSS class, HTML ID, conditions

Each field type has a dedicated input widget: color fields use a combined hex + color-picker input; select fields use a styled dropdown; toggle fields use a switch; URL fields validate and offer media-library picker.

5. Layers Panel (Tree View)

The left "Layers" tab shows the full block tree, indented. Each node shows its type icon + label (or first text prop). You can:

  • Click to select
  • Drag-and-drop to reparent or reorder
  • Toggle visibility eye
  • Duplicate / delete via context menu (right-click or hover icons)

8. Key Design Decisions

Decision 1 — Single source of truth for props

Each block has one props object. No split between data, config, styling. The schema groups them into tabs visually, but storage is flat. This eliminates the current confusion about which JSON blob holds which field.

Decision 2 — BlockRenderer replaces PageSectionRenderer + all hardcoded sections

There will be no more AboutVisionMissionSection.tsx. The "Vision, Mission, Objectives" section becomes a section block containing a card-group block with 3 card children — all fully editable.

Decision 3 — layout_json stored on the page row

A new layout_json TEXT column on pages stores the entire block tree as a single JSON document. The old page_sections table is kept for legacy pages until migration is complete. New pages use layout_json only.

Decision 4 — Migration adapter

On page load, if layout_json is empty but page_sections exist, the builder auto-converts each flat section into a top-level block node. This is 1:1 compatible — no data is lost.

Decision 5 — Backwards compatibility

Public pages that still use old section components continue to work until Phase 7. The builder upgrade is opt-in per page. No forced migration.

10. Questions Before We Start

  1. Backend change approval: Adding layout_json column to pages table requires a DB migration and backend API change to accept/return it. Approved?
  2. Phase order: Do you want to start with the data model + BlockRenderer first (the foundation), or start with the UI (canvas + property panel) using a mock tree in state?
  3. Inline text editing: Should clicking a heading/paragraph on the canvas allow direct in-place editing (contentEditable), or always edit through the right panel?
  4. Drag-and-drop library: Use a library like @dnd-kit/core for robust DnD (handles touch, keyboard, nested containers), or stay with native HTML5 drag events?
  5. Hardcoded pages (About, Board, etc.): Should we migrate them to block trees in Phase 7, or keep them as custom React components permanently?
